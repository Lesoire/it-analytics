# Лабораторна робота №6
## Алгоритмічна сегментація користувачів та аналіз ціннісних орієнтирів

> **Термінологія:**  
> **RFM** (Recency, Frequency, Monetary) — модель сегментації за давністю / частотою / сумою покупок · **ABC** — поділ клієнтів за внеском у дохід (принцип Парето 80/15/5%) · **XYZ** — поділ за стабільністю поведінки · **NTILE** — віконна функція для рівномірного поділу на N груп · **COGS** — прямі операційні витрати ($0.005/подія) · **JTBD** (Jobs To Be Done) — методологія визначення «завдання» користувача · **Running Total** — накопичувальна сума · **Contribution Margin** — валова маржа на рівні користувача

---

## Мета роботи

Опанувати методи автоматизованої класифікації бази користувачів за допомогою RFM та ABC/XYZ моделей, навчитися оцінювати фінансову ефективність кожного сегмента з урахуванням операційних витрат та ідентифікувати поведінкові патерни за методологією JTBD.

---

## Хід роботи

### Блок 1: Алгоритмічна сегментація

**Завдання 1.1 — RFM-скоринг:**

Написано багаторівневий CTE-запит:
1. Агрегація базових метрик по кожному `user_pseudo_id`: дата останньої покупки (`MAX`), кількість транзакцій (`COUNT`), сума доходу (`SUM`)
2. Конвертація дати у кількість днів через `DATE_DIFF` відносно точки відліку 2021-01-31
3. Присвоєння балів 1–5 через `NTILE(5)`. Важливий нюанс для Recency: сортування `DESC` (менше днів = кращий бал = 5), для F і M — `ASC`
4. Конкатенація балів у код сегмента через `CONCAT(CAST(r AS STRING), ...)`
5. Фільтрація трьох ключових сегментів та розрахунок їхньої частки в загальному доході

**Завдання 1.2 — ABC/XYZ аналіз:**

1. ABC — розрахунок накопичувальної частки доходу через `SUM(revenue) OVER(ORDER BY revenue DESC)`. Порогові значення: A ≤ 80%, B ≤ 95%, C — решта
2. XYZ — стабільність через кількість місяців активності: X = 3 місяці, Y = 2, Z = 1
3. Перетин з RFM: підрахунок користувачів що одночасно є Champions (555) і класу A через `INNER JOIN`

### Блок 2: Value-based та JTBD сегментація

**Завдання 2.1 — Маржинальний аналіз:**

Розраховано Contribution Margin = revenue − (event_count × $0.005). Синтетична вартість $0.005/подія моделює реальні витрати на обробку API, логування та зберігання в хмарі. Виділено сегмент Negative Margin Users — користувачі чиї витрати на інфраструктуру перевищують принесений дохід.

**Завдання 2.2 — JTBD класифікація:**

Через `COUNTIF` агреговано поведінкові метрики (view_item, add_to_cart, purchase) і застосовано `CASE WHEN` логіку для присвоєння "роботи" кожному користувачу.

## Результати

### 1.1 RFM-сегментація та скоринг

| Сегмент | Користувачів | % доходу |
|---------|-------------|----------|
| Champions (555) | 49 | 2.57% |
| At Risk (211–255) | 167 | 3.16% |
| Hibernating (111) | 36 | 0.11% |

**Аналіз:**

Сегмент At Risk дає вищу частку доходу ніж Champions (3.16% проти 2.57%). Це пояснюється тим, що Champions визначається через строгу рівність «555» (вузька умова = менше людей), тоді як At Risk охоплює ширший діапазон кодів (211–255). Попри меншу кількість, Champions — найцінніший сегмент з точки зору лояльності та довгострокового LTV.

**Продуктові дії:**
- **Champions:** VIP-програма утримання, пріоритетна підтримка, ранній доступ до нових фіч, окремий rate-limit bucket в API (без throttling)
- **At Risk:** win-back кампанія є найвищим бізнес-пріоритетом — ці клієнти вже доводили готовність платити, їх втрата безпосередньо знижує revenue
- **Hibernating:** реактиваційна кампанія з мінімальними витратами або виключення з дорогих каналів комунікації

### 1.2 ABC/XYZ-матриця та перетин з RFM

**Перетин Champions (555) + клас A: 45 користувачів** — абсолютний пріоритет для інженерної команди.

| ABC/XYZ | Користувачів | Дохід |
|---------|-------------|-------|
| A/X | 2 | 631 |
| A/Y | 131 | 26 702 |
| A/Z | 1 757 | 262 270 |
| B/Y | 19 | 970 |
| B/Z | 1 158 | 52 177 |
| C/Y | 3 | 66 |
| C/Z | 1 349 | 19 349 |

**Аналіз:**

Домінуючий сегмент — **A/Z**: 1 757 користувачів з високим доходом але нерегулярною активністю (лише 1 місяць з 3). Це типова картина для e-commerce без підписочної моделі — покупки відбуваються ситуативно, а не за звичкою.

Клас **A/X** (лише 2 користувачі) — найстабільніші VIP-платники. Їхні дані мають найвищий пріоритет резервування: у разі інциденту відновлення починається з цього сегмента. Клас **C/Z** — "хвіст" бази, де витрати на зберігання та обробку подій потенційно перевищують принесений дохід.

**Технічна пріоритизація для класу A:** без throttling в API, підвищений SLA відновлення після збою (< 1 год), участь у beta-тестуванні, окремий кеш-шар для персоналізованих рекомендацій.

### 2.1 Маржинальний аналіз (COGS = $0.005/подія)

| Метрика | Значення |
|---------|----------|
| Всього користувачів | 270 199 |
| Користувачів з від'ємною маржею | 266 088 (98.49%) |
| Середня валова маржа | $1.26 |
| Мінімальна маржа | -$5.44 |
| Максимальна маржа | $1 526.99 |

**Аналіз:**

98.49% користувачів збиткові — це очікуваний результат для e-commerce, де переважна більшість відвідувачів нічого не купує, але генерує події (перегляди, кліки), які коштують системі ресурсів. Середня маржа $1.26 підтримується позитивною завдяки невеликій кількості платників з високим LTV.

**Стратегія Rate Limiting для Negative Margin Users:**
- Квота ~500 events/day для профілів з margin < 0
- Агресивне кешування відповідей API для "важких" переглядачів без конверсії
- Throttling подій `view_item` протягом 7-денного вікна для збиткових сегментів
- Асинхронний batch-запис некритичних аналітичних подій замість синхронного — знижує пікове навантаження на БД

### 2.2 JTBD-сегментація

| JTBD паттерн | Користувачів | % |
|---------|-------------|---|
| Price Comparison | 3 822 | 32.15% |
| Window Shopping | 8 066 | 67.85% |

**Визначення паттернів:**
- **Price Comparison** — переглянув > 10 товарів, але жодного разу не додав у кошик. Активно досліджує ринок, порівнює пропозиції
- **Window Shopping** — велика кількість подій (≥ 50) без жодної покупки. "Гуляє" по каталогу без конкретного наміру

**Продуктові рішення:**

Для **Price Comparison** (32%): сповіщення "ціна впала на X% від вашого перегляду", бейдж "найнижча ціна за 30 днів", інструмент порівняння двох товарів в один клік — знижує friction між browse і add_to_cart, конвертує дослідника в покупця.

Для **Window Shopping** (68%): персоналізовані добірки на основі переглянутих категорій, обмежені за часом пропозиції з лічильником ("залишилось 3 шт"), сегментований email-retargeting через 24 год після сесії без покупки.

## Загальні висновки

Алгоритмічна сегментація дозволяє перейти від "середнього користувача" до точкових дій з конкретними групами. RFM-модель виявила що лише 49 Champions генерують 2.57% доходу — мала кількість, але критична для утримання. ABC/XYZ показав що 262 270$ (72% доходу) припадає на сегмент A/Z — клієнтів з нерегулярними але великими покупками, що вказує на необхідність підписочної або програмної моделі лояльності для стабілізації доходу.

Маржинальний аналіз підкреслює важливість cost-efficient engineering: 98% бази збиткова на рівні інфраструктури, і без rate limiting "безкоштовне" використання продукту має реальну вартість для компанії. JTBD-класифікація дає конкретні точки для product roadmap замість абстрактного "покращити UX".
