# Лабораторна робота №7
## Аналіз когортної динаміки та моделювання життєвого циклу користувача

> **Термінологія:**  
> **Cohort** — група користувачів об'єднана за місяцем/тижнем першого візиту · **M0/M1/M2** — порядковий місяць когорти з моменту залучення · **W1/W4/W12** — порядковий тиждень · **Retention Rate** — % користувачів що повернулись · **PMF** (Product-Market Fit) — ознака відповідності продукту ринку · **Frown/Smile Curve** — форма кривої retention · **FCM** (Firebase Cloud Messaging) — push-розсилка · **MAU** (Monthly Active Users) — активні за місяць · **Lifecycle State Machine** — алгоритмічна класифікація стану кожного користувача

---

## Мета роботи

Дослідити утримання користувачів у часі через когортні матриці, навчитися діагностувати Product-Market Fit за формою кривої Retention та побудувати Lifecycle State Machine для моніторингу переходів між сегментами бази.

---

## Хід роботи

### Блок 1: Місячний когортний аналіз

**Завдання 1.1 — Побудова матриці утримання:**

Запит реалізовано через 5 CTE:
1. `перший_місяць` — знаходить `MIN(event_date)` для кожного `user_pseudo_id` і трункує до місяця через `DATE_TRUNC(..., MONTH)`
2. `активність` — всі унікальні комбінації user + місяць активності (`SELECT DISTINCT`)
3. `когортна_активність` — `INNER JOIN` між першим візитом і активністю, розрахунок номера місяця через `DATE_DIFF`
4. `розміри_когорт` — кількість унікальних користувачів у M0 (знаменник для %)
5. Фінальний `SELECT` — розрахунок `retention_pct` через `SAFE_DIVIDE`

Перевірка коректності: M0 для кожної когорти завжди = 100%.

**Завдання 1.2 — Аналіз плато та Anti-Churn Trigger:**

На основі матриці з 1.1 аналізується динаміка зміни retention між місяцями. Якщо крива не стабілізується (різниця між M1 і M2 залишається великою) — PMF не досягнуто.

### Блок 2: Тижневий когортний аналіз

**Завдання 2.1 — Weekly Retention:**

Аналогічна логіка але з `DATE_TRUNC(..., WEEK(MONDAY))` замість місяця. Аналіз охоплює 12 тижнів для перших трьох когорт. Результат експортується в CSV для візуалізації через Python/Matplotlib.

### Блок 3: Lifecycle State Machine

**Завдання 3.1 — Класифікація станів:**

Використано метод Flags Matrix — найефективніший підхід у BigQuery:
1. Для кожного `user_pseudo_id` через `MAX(IF(month = 'YYYYMM', 1, 0))` створюються бінарні прапорці активності за листопад, грудень, січень
2. Визначається перший місяць появи (`MIN(event_date)`)
3. `CASE WHEN` логіка присвоює один з чотирьох статусів

## Результати

### 1.1 Місячна когортна матриця

| Когорта | M0 | M1 | M2 |
|---------|----|----|-----|
| 2020-11 | 100% | 5.86% | 1.52% |
| 2020-12 | 100% | 2.53% | — |
| 2021-01 | 100% | — | — |

**Аналіз M1: листопад vs грудень:**

Листопад M1 = 5.86%, грудень M1 = 2.53% — різниця у 2.3 рази. Грудень — місяць святкових розпродажів (Black Friday, Cyber Monday), що дає різкий приплив нових користувачів. Однак ці користувачі прийшли за знижкою, а не через реальний інтерес до продукту — звідси значно гірше утримання. Це класичний trade-off acquisition: більший об'єм залучення за рахунок нижчої якості когорти.

---

### 1.2 Аналіз PMF та Anti-Churn механізм

**Стан плато:**
- M2 листопадової когорти = **1.52%** (суттєво нижче порогу 10%)
- PMF **не досягнуто**: крива падає M0 (100%) → M1 (5.86%) → M2 (1.52%) без ознак стабілізації
- Форма кривої — **Frown Curve** (безперервне спадання), а не Smile Curve (стабілізація або зростання на пізніх місяцях)

Відсутність плато означає, що продукт не формує звичку у користувачів — вони не повертаються самостійно без зовнішнього стимулу.

**Технічний Anti-Churn Trigger:**

```sql
-- Логіка для щоденного batch-job (cron)
IF last_active_at < CURRENT_DATE - 14
   AND user_cohort IN ('2020-11', '2020-12')
THEN
  UPDATE users SET lifecycle_status = 'at_risk';
  SEND fcm_push(template = 'winback', deep_link = '/deals');
END IF;
```

Вікно 14 днів обрано як "рання зона ризику" — втручання до того, як відтік стає незворотним. Пріоритет — листопадова когорта (M1 = 5.86%) як якісніша база для реактивації.

### 2.1 Тижневі криві Retention

| Когорта | W1 | W4 | W12 |
|---------|-----|-----|------|
| 2020-10-26 | 10.82% | 1.35% | 0.25% |
| 2020-11-02 | 6.33% | 2.28% | 0.41% |
| 2020-11-09 | 6.73% | 2.46% | — |

**Аналіз:**

Жодна когорта не стабілізується вище 5% — плато відсутнє на всьому 12-тижневому горизонті. До W12 retention становить 0.2–0.4%. Є незначні коливання на W8–W10, але без стійкого відновлення — це статистичний шум, а не Smile Curve.

**Висновок щодо PMF:** продукт не формує habit loop — регулярної потреби повертатися. Для виправлення Frown Curve необхідний швидший Aha-moment (перший `view_item` в межах 60 секунд після входу), спрощений checkout в один крок, wishlist із push-нагадуванням при зниженні ціни, персоналізований onboarding.

### 3.1 Lifecycle State Machine — стани на 31 січня 2021

| Статус | Визначення | Користувачів | % |
|--------|-----------|-------------|---|
| Churned | Був у грудні, зник у січні | 101 106 | 51.61% |
| New | Вперше прийшов у січні | 91 069 | 46.49% |
| Retained | Був у грудні І в січні | 3 209 | 1.64% |
| Resurrected | Не було в грудні, повернувся в січні | 512 | 0.26% |

Загальна кількість: **195 896** користувачів (активні в грудні або січні).

**Аналіз здоров'я бази:**

Retained = лише 1.64% — критично мале "ядро" лояльних. Здорова база зазвичай має Retained > 10–15%. Структура бази: більше половини — відтік (51.6%), майже половина — нові без гарантії утримання (46.5%).

**Resurrected vs New:** 512 vs 91 069 = 0.56% від нових. Це свідчить про слабкий Memory Value — продукт не залишається у пам'яті після паузи. Можлива технічна причина: скидання `user_pseudo_id` при очищенні cookies призводить до того, що система не впізнає тих хто повернувся і записує їх як нових, занижуючи реальну кількість Resurrected.

**Баланс за січень:** Inflow (New + Resurrected) = 91 581 < Outflow (Churned) = 101 106 — чиста втрата 9 525 MAU. Це безпосередньо узгоджується з Quick Ratio < 1 у ЛР8.

## Загальні висновки

Когортний аналіз однозначно показує: продукт знаходиться у стані активної деградації бази без ознак Product-Market Fit. Три незалежних індикатори підтверджують це:

1. **Місячний retention**: M2 = 1.52% без ознак плато — Frown Curve
2. **Тижневий retention**: жодна когорта не стабілізується вище 5% за 12 тижнів
3. **Lifecycle**: Retained = 1.64%, Churned = 51.6% — база стрімко "витікає"

Критичний висновок для продуктової команди: масштабування acquisition при такому стані бази є неефективним — кожен новий залучений користувач з великою ймовірністю покине продукт протягом першого місяця. Пріоритет — зупинити відтік через Aha-moment оптимізацію та Anti-Churn тригери для якісних когорт.
