# Лабораторна робота №8
## Growth Accounting: фінансове моделювання росту та проєктування систем утримання

> **Термінологія:**  
> **Quick Ratio** — (Нові + Воскреслі) ÷ Відтік, "спідометр росту" · **MAU** (Monthly Active Users) — активні за місяць · **NRR** (Net Revenue Retention) — збереження доходу від наявної бази · **Expansion Revenue** — приріст доходу від існуючих клієнтів · **Negative Churn** — стан коли NRR > 100% · **Activation Loop** — механізм "дотискання" нових користувачів до Aha-moment · **Resurrection Loop** — система повернення "сплячих" · **JSON Contract** — структура даних для міжсервісної передачі · **Pub/Sub** — асинхронна черга повідомлень

---

## Мета роботи

Опанувати математичний апарат обліку росту продукту (Quick Ratio), навчитися аналізувати фінансовий вплив лояльної бази через NRR та спроєктувати архітектуру автоматизованих циклів активації та повернення користувачів.

---

## Хід роботи

### Блок 1: Quick Ratio та Growth Accounting

**Завдання 1.1–1.2 — Quick Ratio і What-if аналіз:**

Запит побудовано на основі Lifecycle State Machine з ЛР7. Через `COUNTIF` агреговано кількість користувачів у кожному стані. Quick Ratio = `SAFE_DIVIDE(new + resurrected, churned)`. What-if розрахунок: цільовий відтік для QR=1.5 = `(new + resurrected) / 1.5`, відсоток зниження = різниця між поточним і цільовим відтоком.

**Завдання 1.3 — Візуалізація Growth Balance Chart:**

Python-скрипт будує bar chart з двома стовпчиками: Inflow (зелений) та Outflow (червоний), горизонтальна лінія нульового балансу, Quick Ratio у заголовку.

### Блок 2: NRR та Expansion Revenue

**Завдання 2.1–2.3 — NRR розрахунок:**

Ключовий принцип — User-Level Aggregation. Через `SUM(IF(month=12,...))` і `SUM(IF(month=1,...))` зводимо дохід кожного користувача за обидва місяці в один рядок. Retained cohort = ті у кого обидва прапорці активності = 1. Expansion = сума позитивних різниць `GREATEST(jan - dec, 0)`. NRR = retained_jan / total_dec × 100%.

### Блок 3: Retention Loops — System Design

**Завдання 3.1 — Activation Loop:**

Знаходимо нових користувачів січня без `view_item` за перші 24 години через `NOT EXISTS` підзапит для перевірки що це дійсно нові (немає подій до січня), та `COUNTIF` з часовим обмеженням `event_timestamp ≤ first_event_ts + 24h`.

**Завдання 3.2 — Resurrection Loop:**

Для Dormant (active_dec=1, active_jan=0) витягуємо останню переглянуту категорію через `ARRAY_AGG(item_category ORDER BY event_timestamp DESC LIMIT 1)[OFFSET(0)]` після `UNNEST(items)`. Формуємо JSON-контракт для сервісу розсилок.

## Результати

### 1.1 Quick Ratio — "Спідометр росту"

| Метрика | Значення |
|---------|----------|
| New | 91 069 |
| Resurrected | 512 |
| Churned | 101 106 |
| Retained | 3 209 |
| Inflow (New + Resurrected) | 91 581 |
| Outflow (Churned) | 101 106 |
| Чистий приріст MAU | **-9 525** |
| Quick Ratio | **0.9058** |
| Вердикт | **Critical** |

**Аналіз:**

Quick Ratio = 0.9058 < 1.0 — продукт перебуває у стані "Critical". За кожного нового залученого користувача продукт втрачає більше одного існуючого. Чиста втрата активної бази за січень 2021 — 9 525 MAU. На графіку Growth Balance Chart стовпчик Outflow (червоний) перевищує Inflow (зелений).

Практичний висновок: масштабування acquisition при QR < 1 є фінансово неефективним — бюджет на залучення нових компенсує лише частину втрат від відтоку, не забезпечуючи реального росту MAU. Пріоритет — retention та anti-churn заходи.

### 1.2 What-if аналіз: досягнення QR = 1.5

| Метрика | Значення |
|---------|----------|
| Поточний Churned | 101 106 |
| Цільовий Churned при QR=1.5 | 61 054 |
| Необхідне зниження відтоку | **39.61%** |

**Аналіз:**

Для виходу на здоровий ріст (QR = 1.5) при незмінному рівні залучення необхідно знизити відтік на 39.61% — з 101 106 до 61 054 користувачів. Це значна але досяжна ціль через комплекс retention-заходів.

**Інженерні причини надмірного відтоку:** затримка > 3 с на першому екрані (підтверджено в ЛР5: P50 = 5.04 с), збої авторизації, broken deep links після email/push, некоректна обробка помилок при повторному вході, відсутність персоналізованого onboarding.

### 2.1–2.3 NRR та Expansion Revenue

| Метрика | Значення |
|---------|----------|
| Дохід Retained-когорти (грудень) | $26 782 |
| Дохід Retained-когорти (січень) | $11 872 |
| Expansion Revenue (upsell) | $10 289 |
| Contraction Revenue (downsell) | $25 199 |
| Net Expansion | **-$14 910** |
| NRR | **44.33%** |
| Revenue Churn proxy | 92.61% |
| Upsell users | 161 |

**Аналіз:**

NRR = 44.33% — критично низький показник. Це означає що навіть ті користувачі, які залишились (Retained), витратили в січні лише 44 центи на кожен долар, витрачений у грудні. Expansion Revenue ($10 289 від 161 upsell-користувачів) суттєво поступається Contraction ($25 199) — база "стискається" фінансово, а не лише кількісно.

**Negative Churn (NRR > 100%):** найлегше досягається через **підписочну модель** (auto-renew + tier upsell) або usage-based монетизацію з volume discounts — обидва механізми автоматично нарощують дохід від існуючих клієнтів без їхнього активного рішення. При NRR > 100% бізнес може рости навіть без залучення нових клієнтів.

**Фіча для upsell:** рекомендаційний блок "Разом з цим купують" на сторінці оформлення + bundle-знижка для Retained-сегменту — стимулює Expansion Revenue від найлояльніших користувачів.

### 3.1–3.3 Activation та Resurrection Loops

| Метрика | Значення |
|---------|----------|
| Нових users (січень) | 91 069 |
| Без view_item за перші 24 год | 73 623 |
| % activation gap | **80.84%** |
| Dormant users | 101 106 |

**Activation Loop — логіка тригера:**

80.84% нових користувачів не досягли Aha-moment (перший перегляд товару) за перші 24 години. Це критичний показник — саме в цьому вікні формується або не формується звичка повертатися.

Алгоритм:
1. Подія `session_start` → стартує таймер у backend
2. Через 24 год система перевіряє: чи є `view_item` для цього `user_pseudo_id`?
3. Якщо ні → відправляємо FCM push з бестселерами тижня + купон на першу покупку
4. Якщо так → скасовуємо тригер (користувач вже активований)

**Resurrection Loop — JSON контракт:**

Для Dormant користувачів витягується остання переглянута категорія і формується персоналізований payload:

```json
{
  "user_id": "1242944.3150378184",
  "segment_type": "resurrection",
  "last_category_name": "Lifestyle/Bags/",
  "personalized_message": "Ми сумуємо за вами! Перегляньте нові надходження у категорії Lifestyle/Bags/"
}
```

**Архітектура системи:**
Cloud Function (тригер: daily cron через Cloud Scheduler) → читає аналітику з BigQuery → записує сегменти в Firestore колекцію `user_segments` → Pub/Sub публікує події → FCM відправляє персоналізований payload на основі `last_category_name`.

**Технічна гіпотеза щодо 80.84% activation gap:** причиною може бути висока latency (підтверджена в ЛР5 — P50 = 5.04 с), баги в onboarding flow, некоректне відображення каталогу на мобільних пристроях, або просто відсутність будь-якого onboarding — користувач потрапляє на порожній екран без підказки "що робити далі".

## Загальні висновки

ЛР8 зводить воєдино всі попередні аналізи у фінансову і операційну картину здоров'я продукту:

**Кількісно:** QR = 0.9058 (Critical), NRR = 44.33%, MAU -9 525/місяць — три незалежні метрики підтверджують одне: продукт перебуває у фазі деградації.

**Причинно:** виявлені в ЛР5 технічні проблеми (SLA 87% сесій > 3 с) безпосередньо впливають на activation gap (80.84%) і відтік. Відсутність PMF з ЛР7 знаходить фінансове відображення в NRR < 50%.

**Операційно:** спроєктовані Activation та Resurrection Loops дають конкретний технічний план — не "покращити UX", а конкретні тригери з чіткою логікою, JSON-контрактом та архітектурою. Якщо activation gap знизити з 80% до 50% — це додатково 27 000+ активованих нових користувачів на місяць, що безпосередньо покращить Quick Ratio.
